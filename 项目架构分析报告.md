## 二、当前架构判断

### 2.1 后端架构

当前后端仍然是典型的单体分层结构：

```text
Controller -> Service -> Mapper -> MySQL
```

主要特点：

- `Controller` 层负责 HTTP 接口和部分参数预处理
- `Service` 层承载主要业务逻辑
- `Mapper` 层通过 MyBatis Plus 和 XML Mapper 访问数据库
- `Entity` 直接参与数据库映射，DTO 仅在少量场景使用
- 使用 `Result` 统一返回格式
- 使用 `GlobalExceptionHandler` 做全局异常兜底
- 使用 `JwtInterceptor` 做登录态校验
- 使用 `ScheduledTasks` 做 OneNET 定时同步
- 使用 `Actuator` 提供基础健康检查

结论：

- 架构清晰，适合课程设计和中小型单体项目
- 业务边界基本明确，但控制层仍存在业务逻辑渗透
- 安全和工程化能力明显弱于业务功能实现

### 2.2 前端架构

前端是 Vue 2 单页应用，采用页面组件 + 路由 + Vuex + Axios 的常规结构。

```text
Views -> Components -> Router / Vuex -> request.js -> Backend API
```

主要特点：

- 页面和组件拆分较多，功能模块完整
- 路由大量使用动态 `import()`，懒加载做得并不差
- 路由菜单根据后端返回的 `menus` 动态注册
- 登录态、菜单和部分 UI 状态大量依赖 `localStorage`
- `Vuex` 仅承担少量全局状态，不是主要状态中心

结论：

- 前端功能覆盖面较大，适合演示型项目
- 真实生产可维护性一般，主要问题是状态管理分散、日志过多、鉴权依赖本地存储

## 四、旧版报告中需要纠正的结论

### 4.1 已经过时的结论

#### 1. `QWEN` 和 `AMAP` 仍然硬编码

这个结论现在不成立。

当前实际情况：

- `QWEN_API_KEY` 通过环境变量读取
- `AMAP_JS_KEY`、`AMAP_WEB_KEY`、`AMAP_JS_SECURITY_KEY` 通过环境变量读取
- 前端高德地图通过 `VUE_APP_AMAP_JS_KEY` 和 `VUE_APP_AMAP_SECURITY_CODE` 读取
- OneNET 通过 `ONENET_AUTHOR_KEY`、`ONENET_USER_ID`、`ONENET_PRODUCT_ID`、`ONENET_DEVICE_NAME` 等环境变量读取

#### 2. 全局异常处理不完整到“基本缺失”

这个结论不准确。

当前实际情况：

- 已有 `GlobalExceptionHandler`
- 已处理业务异常、参数绑定异常、数据库异常、上传大小异常、空指针和通用异常

更准确的说法应该是：

- 全局异常处理已经存在
- 但控制层普遍没有系统使用 `@Valid / @Validated`
- 因此很多输入校验仍停留在手写 `if` 判断层面

#### 3. 缺少定时任务管理

这个结论不准确。

当前实际情况：

- 已存在 `ScheduledTasks`
- 使用 `@Scheduled(fixedRate = 30000)` 每 30 秒同步 OneNET 数据

更准确的说法应该是：

- 定时任务已经存在
- 但任务频率、设备范围和执行逻辑写死在代码中，缺乏可配置性和运维友好性

#### 4. 缺少监控能力

这个结论不准确。

当前实际情况：

- 已引入 `spring-boot-starter-actuator`
- 拦截器已放行 `/actuator/**`

更准确的说法应该是：

- 已有基础健康检查能力
- 但没有 Prometheus / Grafana / 日志聚合 / 报警闭环

#### 5. 路由懒加载不足

这个结论不准确。

当前实际情况：

- 登录、注册、首页、地理地图、3D 地图、经营分析、库存、销售等页面均采用 `() => import(...)`
- 路由懒加载整体是到位的

真正的问题是：

- 路由依赖 `menus` 动态拼接
- 菜单和路由恢复逻辑基于 `localStorage`
- 这一套机制在异常恢复和权限一致性方面较脆弱

#### 6. 存在“全局 SQL 注入高风险”

旧版表述过度。

当前从代码看：

- 大量查询使用 `QueryWrapper` / MyBatis Plus
- 未看到明显的字符串拼接式原生 SQL 组装

更准确的说法应该是：

- 目前未发现明确的高风险 SQL 拼接点
- 但仍应继续审查 XML Mapper、模糊查询和动态条件写法

## 六、当前最值得优先处理的问题

### 6.1 P0：账号与认证安全

这是当前最需要优先处理的部分。

1. 密码明文存储和明文比对
2. 默认密码和重置密码固定为 `123`
3. JWT 直接绑定用户密码生成与校验
4. 前端把 token 放在 `localStorage`

如果只做一轮安全整改，优先级建议如下：

1. 引入密码哈希，如 BCrypt
2. 重构 JWT 签名密钥，不再使用用户密码
3. 重置密码改为一次性随机密码或短信/邮箱流程
4. 减少 `localStorage` 暴露面，至少统一封装认证状态

### 6.2 P1：清理调试代码和日志

当前有大量演示期和排障期遗留输出：

- 后端 `System.out.println`
- 前端大量 `console.log`

问题不只是“代码不整洁”，还包括：

- 噪音大
- 泄露内部状态
- 生产问题排查反而更困难

### 6.3 P1：参数校验和 DTO 规范

当前控制层很多接口是：

- 直接接收 Entity
- 手写空值判断
- 缺少统一校验注解

建议后续逐步改成：

- DTO 入参
- `@Valid / @Validated`
- 明确字段范围、格式、长度约束

### 6.4 P2：工程化能力补齐

目前可以跑，但工程化能力偏弱：

- 测试只有最基础的 `contextLoads()`
- 无缓存层
- 无接口限流
- 无集中监控
- 无统一前端错误上报

这些不影响演示，但会直接限制后续稳定性。

## 七、升级建议

### 7.1 短期建议

- 保留单体架构，不急于拆微服务
- 优先修安全问题和调试输出
- 把参数校验规范化
- 给关键模块补最基础的单元测试和接口测试

### 7.2 中期建议

- 引入 Redis 做热点数据缓存
- 增加接口限流
- 给 OneNET 同步任务增加开关和配置化参数
- 增加统一日志规范和错误追踪

### 7.3 长期建议

- Spring Boot 升级到 2.7.x 或 3.x 前，先规划 JDK 升级和 Jakarta 迁移
- Vue 2 迁移 Vue 3 前，先清理 `localStorage` 依赖和页面级混乱逻辑
- 如果后续真有多设备、多算法、多租户需求，再评估服务拆分

## 八、修正后的结论

### 8.1 这份项目当前“说得对”的部分

- 项目确实是分层单体架构
- 前端确实是 Vue 2 组件化结构
- JWT、安全、默认密码、调试输出确实存在明显问题
- 缓存、限流、测试覆盖和监控体系确实偏弱

### 8.2 这份项目当前“说得不对”的部分

- `QWEN`、`AMAP` 和 `OneNET` 主要敏感配置已不再是源码硬编码
- 并不是没有全局异常处理
- 并不是没有定时任务
- 并不是完全没有监控能力
- “路由懒加载不足”不成立
- “全局 SQL 注入高风险”缺少证据，原表述过重
- 环境变量清单缺少前端高德变量，且误写了当前并未实际接入的 OneNET 环境变量

### 8.3 当前最准确的一句话总结

这是一个功能完整、展示效果强、业务覆盖面广的智慧农场单体项目；当前主要短板不在“有没有架构”，而在“安全基线、状态管理、调试遗留和工程化质量”。
